# 面试指南代码规范

## 概述

将此技能视为本仓库每次代码变更的必备要求。

保持代码有明确意图、类型完备且可观测。优先优化可维护性，而不是微优化，除非已有明确证据表明后者必要。

## 工作流程

1. 在编辑前先判断改动类型：`后端`、`前端`、`跨层`、`仅测试` 或 `影响行为的配置改动`。
2. 在编写代码前阅读相关参考文件：
   - 后端 Java/Spring: [references/java-backend.md](references/java-backend.md)
   - 前端 TypeScript/React: [references/frontend-typescript.md](references/frontend-typescript.md)
   - 验证、日志记录和最终检查清单: [references/testing-and-observability.md](references/testing-and-observability.md)
3. 应用通用约束：
   - 保持 `app/src/main/java/interview/guide` 和 `frontend/src` 下现有的边界
   - 优先使用类型化的请求/响应模型，而不是临时映射、原始集合、`any` 或无限制的 `Object`
   - 保持控制器和页面精简；将业务逻辑移入服务、Hook 或职责单一的辅助函数中
   - 返回空集合而不是 `null`
   - 在状态转换、重试、外部调用和失败周围添加或更新日志
   - 对每个行为更改添加或更新测试或构建验证
4. 仅运行与更改文件相关的验证，然后报告已运行和未运行的内容。

## 共享规则

- 每个文本文件使用 UTF-8。
- 保持修改范围窄。不要混合重构、格式调整和行为更改，除非任务需要。
- 优先使用小的可组合方法，而不是具有隐藏副作用的长过程。
- 优先使用明确的领域词汇，而不是 `data`、`info`、`util`、`handler` 或 `temp` 等通用名称。
- 如果可以命名稳定模型，就不要引入新的弱类型公共结构。
- 将触及的代码逐步收敛到更强的类型和更清晰的契约，但不要为了满足规则而重写无关区域。
- 当周边风格已经一致时就遵循现有风格；只有在局部区域不清晰或容易出错时才改进。

## 选择规则

用更清晰的选项，而不是更聪明的那个。

- Java `record`：
  用于不可变 DTO、查询/结果模型，以及具备值语义的内部载荷。
  JPA 实体、可变状态承载对象或 Spring 组件使用 `class`。
- Java 循环与流：
  对纯粹的过滤、映射、收集和聚合使用 Stream。
  当逻辑包含分支、状态修改、重试或日志时，使用循环。
- TypeScript `interface` 与 `type`：
  对表示稳定契约、后续可能演进或合并的导出对象结构使用 `interface`。
  对联合类型、别名、元组和派生辅助类型使用 `type`。
- DTO 与 `Map<String, Object>`：
  新的 API 契约和跨层返回值优先使用 DTO。
  只有小型、本地、以兼容性为导向的胶水代码才允许使用 `Map`。
- `Optional` 与可空值：
  对表示“可能缺失”的 Java 返回值使用 `Optional`。
  不要在实体字段、DTO 字段或方法参数中使用 `Optional`。
- 异常与状态标志：
  对异常流程抛出领域异常。
  除非调用者真的只需要成功或失败，否则不要用布尔返回值隐藏失败信息。

## 最终检查

在结束前请确认以下所有内容：

- 名称符合仓库惯例并反映域名意图
- 更改的文件格式化为本地样式
- 未引入新的原始类型、`any`、静默 `catch` 或返回 `null` 集合
- 日志包含标识符，避免秘密或大型负载
- 测试或构建验证涵盖了已更改的行为
- 最终响应提到应用了哪些参考文件以及运行了哪些验证
